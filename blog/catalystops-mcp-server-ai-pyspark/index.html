CatalystOps Now Has an MCP Server — AI-Assisted PySpark Optimization

March 13, 2026  ·  6 min read

CatalystOps 0.8.2 ships a built-in MCP (Model Context Protocol) server. Claude, GitHub Copilot, Cursor, and any other MCP-compatible AI assistant can now read your live Spark analysis results, trigger dry runs against your Databricks cluster, and pull billing data — all from inside your editor, without copy-pasting a single line.

What is MCP?

The Model Context Protocol is an open standard, introduced by Anthropic, that lets AI assistants connect to external tools and data sources in a structured way. Instead of pasting your code into a chat window and asking "what's wrong with this?", an MCP-enabled AI can reach directly into your running tools — IDEs, databases, APIs — and pull live context.

Think of it as a USB-C standard for AI tool integrations. Claude has MCP support built in. So does GitHub Copilot (VS Code 1.99+), Cursor, and Windsurf. Once a tool exposes an MCP server, any of these AI clients can use it immediately — no per-integration glue code required.

What the CatalystOps MCP Server Exposes

When CatalystOps starts, it spins up a local HTTP server on 127.0.0.1 and writes a .vscode/mcp.json file so VS Code-based AI clients auto-discover it. No manual configuration needed in most setups.

The server exposes three types of objects:

Tools (actions the AI can invoke)

analyze_pyspark
Run local static analysis on any PySpark snippet. Returns issues with severity, line number, description, and fix suggestion. No Databricks connection required.
get_active_file_issues
Returns CatalystOps findings for the file currently open in VS Code — the same diagnostics shown as squiggles in the editor, but as structured JSON.
run_dry_run
Triggers a full Databricks dry run on the active file. Waits up to 5 minutes for results, then returns the physical plan, logical plan, and all plan-level issues (SortMergeJoin, shuffle count, repeated scans, etc.).
get_plan_analysis
Returns parsed Catalyst plan issues from the last dry run without triggering a new one. Useful for asking follow-up questions about an existing result.
get_billing_summary
Returns Databricks spend data (total dollars, DBUs, by user, by job, by workload type) for the past day, week, or month — from the local 1-hour cache.
refresh_billing
Forces a live billing fetch from Databricks, bypassing the cache. Updates the billing tree view in VS Code simultaneously.

Resources (data the AI can read)

  • catalystops://issues/current — live Markdown-formatted issue list for the active file
  • catalystops://plans/last — raw physical and logical plan text from the last dry run
  • catalystops://billing/summary — formatted billing snapshot for the most recent cached period

Prompts (pre-built conversation starters)

  • pyspark_code_review — injects CatalystOps local findings + plan issues as context, then asks the AI to review the code holistically
  • optimize_spark_plan — injects the raw Catalyst plan text and asks the AI to identify join strategy improvements, shuffle reduction, caching opportunities, and predicate pushdown gaps

A Real Example

Here is what a conversation with Claude looks like once CatalystOps is connected via MCP:

You: "Review my PySpark ETL file and tell me what will be expensive at scale."

Claude calls get_active_file_issues to read the static analysis results already sitting in CatalystOps, then calls run_dry_run to trigger a live execution against your Databricks serverless compute. A minute later it comes back with: 

Your pipeline has 3 issues worth addressing at scale:

1. SortMergeJoin on `events` x `users` (line 42)
   At 100GB+, this will trigger a full shuffle on both sides.
   Consider broadcasting `users` if it stays under ~200MB.
   Fix: df.join(broadcast(users_df), "user_id")

2. Repeated scan of `events` table — read 3 times without caching (lines 18, 31, 44)
   Each action re-executes the full lineage from source.
   Fix: events_df.cache() before line 18, unpersist() after line 44.

3. Window without partitionBy on line 67
   Window.orderBy() without partitionBy forces all data into a single partition.
   At scale this will OOM your executors.
   Fix: Window.partitionBy("user_id").orderBy("event_time")

No copy-pasting, no context switching. The AI has access to both the static analysis and the live Catalyst plan — the combination that makes findings actionable.

How It Works Technically

CatalystOps starts a local HTTP server on 127.0.0.1 using an OS-assigned port on every extension activation. The server implements the Streamable HTTP MCP transport in stateless mode — each JSON-RPC request gets a fresh server instance, which avoids state corruption across sessions.

On startup, it writes a .vscode/mcp.json file into your workspace: 

{
  "servers": {
    "CatalystOps": {
      "type": "http",
      "url": "http://127.0.0.1:54321/mcp"
    }
  }
}

VS Code 1.99+ picks this up automatically — GitHub Copilot will see CatalystOps as an available MCP server without any manual setup. For Claude Desktop or other clients, copy the URL from the CatalystOps Output panel and add it to your client's server list.

Setup

Install or update CatalystOps to 0.8.2:

ext install CatalystOps.catalystops

The MCP server starts automatically. In VS Code 1.99+ with GitHub Copilot or Cursor, it is auto-discovered. For Claude Desktop, add the server URL from the Output panel to your claude_desktop_config.json.

To disable the MCP server:

// settings.json
"catalystops.mcp.enabled": false

Why This Matters for PySpark Development

The pattern that MCP enables is fundamentally different from asking an AI to review a code snippet in chat. When an AI has tool access to your live analysis results, it can give advice that is grounded in what your actual Spark plan looks like — not generic best practices based on the code alone.

Static analysis tells you what patterns are present. Runtime plan analysis tells you what those patterns will actually do. MCP lets an AI assistant combine both signals and give you advice that is specific to your pipeline, your data, and your cluster configuration.

This is the "code + plan + cost" combination described in our earlier post on billing visibility — except now it is available to any AI assistant that speaks MCP.
