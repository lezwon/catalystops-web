Bundle Tasks. Bundle Linting.
Databricks Asset Bundle Support in VS Code

CatalystOps 0.12.0 understands your databricks.yml. It lists every task in the sidebar, validates your bundle YAML with 40+ schema-aware rules as you type, and pre-fills connection details from your bundle targets — no URL to copy-paste.

The Problem with DAB Workflows Today

Databricks Asset Bundles (DAB) are the standard way to define and deploy Databricks jobs, pipelines, and ML resources as code. A databricks.yml file declares jobs with tasks pointing to Python scripts or notebooks, cluster configs, target environments, and permissions.

But until now, working with a bundle in VS Code meant a lot of manual context-switching: finding which Python file a task points to, hunting down the right workspace host for a target environment, and catching YAML schema mistakes only after databricks bundle validate fails in the terminal.

CatalystOps 0.12.0 closes that gap entirely — directly inside the editor.

Bundle Tasks Panel

When CatalystOps detects databricks.yml in your workspace root, a Bundle Tasks section appears in the CatalystOps sidebar. It lists every task across the main bundle file and all files referenced by include: patterns:

  • spark_python_task entries show the Python file path relative to the bundle root
  • notebook_task entries with local paths show the notebook path
  • Click any task to open its source file in the editor
  • Use the inline Analyze button to run a dry-run plan analysis on that task's script — no need to navigate to the file first

The panel refreshes automatically whenever databricks.yml or any included resource YAML changes.

💡 No Databricks CLI required

CatalystOps parses the bundle YAML file-side only. You don't need the Databricks CLI installed for the sidebar or linting features — only for actual bundle deploy/run operations.

40+ Inline YAML Validation Rules

CatalystOps validates your bundle YAML as you type. Diagnostics appear as underlines in the editor — no terminal command needed. The rules cover the full bundle schema including resource types, enum fields, cluster configuration, and Databricks-specific quirks that the generic YAML language server doesn't know about.

Alert v2 Schema

The Databricks Alert v2 API schema differs significantly from other resources, and the mistakes are easy to make:

MistakeCorrect form
Using condition: keyUse evaluation:
subscriptions: at alert levelMove under evaluation.notification.subscriptions
schedule.cron_schedule:Use schedule.quartz_cron_schedule:
Missing schedule.pause_statusRequired field — add PAUSED or UNPAUSED

These are all surfaced as errors with a description of the correct form — not just a generic "invalid field" message.

Permission Level Checks

Different resource types use different permission level sets — a common source of deploy failures:

  • Jobs: CAN_VIEW, CAN_MANAGE_RUN, CAN_MANAGE
  • Dashboards / Alerts: CAN_READ, CAN_RUN, CAN_EDIT, CAN_MANAGE
  • Volumes: use grants: not permissions: — CatalystOps errors if permissions: is found on a volume resource

Mutual Exclusion and Cluster Checks

CatalystOps also catches configuration conflicts that the Databricks CLI only reports at validate time:

  • schedule: + continuous: on the same job — error
  • job_cluster_key: + existing_cluster_id: on the same task — error
  • data_security_mode: NONE when a production target exists — warning
  • spark_version format validation (14.3.x / 14.3.x-scala2.12 pattern)
  • environments[].spec.client is required — use "4" for current serverless
  • File existence checks for spark_python_task.python_file and local notebook paths

Bundle Target in Connection Wizard

When databricks.yml is present and defines workspace targets, the CatalystOps: Configure Databricks Connection wizard shows a Use Bundle Target option at the top of the host selection step.

Select a target (dev / prod / staging) and the wizard pre-fills the workspace host from the YAML — no URL to type or copy from the Databricks console.

MCP: list_bundle_tasks

The new list_bundle_tasks MCP tool exposes bundle task metadata to Claude, GitHub Copilot, Cursor, and any MCP-compatible client:

You: What tasks are defined in my bundle?
Claude: ↳ calling list_bundle_tasks…
        Found 3 tasks in 2 jobs:
        - daily_etl → extract_data (spark_python_task): src/extract.py
        - daily_etl → transform (spark_python_task): src/transform.py
        - scoring_job → score (notebook_task): src/score.py

The tool returns the bundle name, all target hosts, and the full task list with job name, task key, type, and file path. Useful for asking an AI assistant to analyze a specific bundle task by name.

Getting Started

  1. Install CatalystOps from the VS Code Marketplace (v0.12.0 or later)
  2. Open a workspace folder that contains databricks.yml
  3. The Bundle Tasks section appears automatically in the CatalystOps sidebar
  4. Open databricks.yml — YAML linting activates immediately, no configuration needed
  5. Run CatalystOps: Configure Databricks Connection — select Use Bundle Target to pre-fill your workspace host
